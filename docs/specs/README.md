# NimbusArc Specs

Create one folder per feature when work is large enough to need a spec.

Recommended structure:

- `docs/specs/<feature>/spec.md`
- `docs/specs/<feature>/plan.md`

Use specs for:

- major UI workflow changes
- scenario engine behavior
- provider model changes
- Terraform, CLI, or LocalStack integrations
- persistence or learner progress changes

## Project Maps

- Product scope and current baseline: `docs/product-overview.md`
- Current and planned capabilities: `docs/feature-map.md`
- Phase implementation roadmap: `docs/roadmap.md`
- Technical boundaries and provider strategy: `docs/architecture.md`
- Security and cost simulation contracts: `docs/security-cost-model.md`
